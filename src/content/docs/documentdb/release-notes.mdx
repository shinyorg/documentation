## 5.2.2 - May 30, 2026
<RN type="fix">**SQL Server `CreateIndexAsync` emitted broken DDL** â€” the `jsonPath` argument was ignored; every JSON-path index registration silently produced a useless index on the `TypeName` column. Indexes are now backed by a persisted computed column (`cc_{indexName}`) over `JSON_VALUE(Data, '$.path')`, with a filtered `CREATE INDEX` on that column. `DropIndex` now drops both the index and its backing computed column using the required `DROP INDEX â€¦ ON [table]` syntax</RN>
<RN type="fix">**Upsert deep-merge was shallow on PostgreSQL and SQL Server** â€” Postgres used the `jsonb || jsonb` concat operator (top-level only) and SQL Server used a flat `OPENJSON â€¦ FULL OUTER JOIN`, both of which clobbered nested objects. Neither database has a native RFC 7396 `JSON_MERGE_PATCH`. Both providers now perform a row-locked read-merge-write fallback in C# (using `SELECT â€¦ FOR UPDATE` on PG and `WITH (UPDLOCK, HOLDLOCK)` on SQL Server), keeping the documented RFC 7396 deep-merge semantics</RN>
<RN type="fix">**Null-stripping was shallow across all providers** â€” when an Upsert patch contained a nested object whose other properties were null (e.g. `new Doc { Address = new Address { City = "X" } }`), the unfilled `street`/`state` nulls reached the merge step and were interpreted as RFC 7396 deletions, silently wiping the stored values. Null-stripping is now recursive, so partial nested patches preserve unspecified fields on SQLite, MySQL, LiteDB, IndexedDB, Cosmos DB, PostgreSQL, and SQL Server</RN>
<RN type="fix">MySQL â€” `DropIndexAsync` emitted `DROP INDEX {name};`, which is invalid in MySQL. Now emits `DROP INDEX {name} ON \`{table}\`;`</RN>

## 5.2.1 - May 29, 2026
<RN type="feature">**Unit of Work** â€” new `CreateUnitOfWork()` extension method on `IDocumentStore` returns a `UnitOfWork` that buffers `Add`/`Update`/`Remove` operations and applies them atomically inside a single transaction on `Commit()`. The queue is auto-cleared on successful commit; on failure the transaction is rolled back and the queue is preserved for inspection or retry. Works across every provider via `RunInTransaction`. See [Unit of Work](/documentdb/indexes#unit-of-work)</RN>

## 5.2 - May 27, 2026
<RN type="feature">**Shiny.DocumentDb.IndexedDb** is now 100% AOT/reflection-free. The JS interop layer was rewritten to use `[JSImport]` from `System.Runtime.InteropServices.JavaScript` instead of `IJSRuntime.InvokeAsync`. The library no longer requires `JsonSerializerIsReflectionEnabledByDefault=true` to function â€” apps targeting AOT or trim-safe deployments can use the IndexedDB provider without re-enabling reflection</RN>
<RN type="feature">Internal source-generated `JsonSerializerContext` (camelCase) for `DocumentRecord` wire-format serialization â€” the library no longer depends on the consuming app's `JsonSerializerOptions` for envelope types. Existing IndexedDB databases remain readable; the wire format is unchanged</RN>
<RN type="feature">Module loading switched from `IJSRuntime.InvokeAsync<IJSObjectReference>("import", ...)` to `JSHost.ImportAsync(...)`. No app-side changes required</RN>
<RN type="fix">`IndexedDbDocumentStore` no longer throws `JsonSerializerIsReflectionDisabled` when the host app has reflection disabled. In 5.1 and earlier, the library's reliance on Blazor's `IJSRuntime` `Object[]` arg envelope forced apps targeting AOT to either drop the IndexedDB provider or globally re-enable reflection</RN>

## 5.1 - May 26, 2026
<RN type="fix">Query operations (`Count`, `Any`, `ToList`, `ToAsyncEnumerable`, `ExecuteDelete`, `ExecuteUpdate`, `Max`, `Min`, `Sum`, `Average`, and projected `Select` queries) now initialize the type-specific table before executing. Previously, the query path always initialized the default `TableName`, so calling a query method against a type registered with `MapTypeToTable<T>()` before any insert had created its table raised `no such table: <Name>`</RN>
<RN type="fix">SQLite â€” table identifiers are now properly quoted in all generated DDL/DML. Mapping a type whose name collides with a SQL reserved word (e.g. `Order`, `Group`, `User`) no longer produces `syntax error` at table creation or insert time</RN>

## 5.0 - May 6, 2026
<RN type="breaking">`Backup` removed from `IDocumentStore` interface â€” now available only on concrete types: `SqliteDocumentStore.Backup()`, `SqlCipherDocumentStore.Backup()`, and `LiteDbDocumentStore.Backup()`</RN>
<RN type="breaking">Provider-specific DI extension methods removed (`AddSqliteDocumentStore`, `AddSqlCipherDocumentStore`, `AddSqlServerDocumentStore`, `AddMySqlDocumentStore`, `AddPostgreSqlDocumentStore`, `AddLiteDbDocumentStore`, `AddCosmosDbDocumentStore`, `AddIndexedDbDocumentStore`). Use `AddDocumentStore` from `Shiny.DocumentDb.Extensions.DependencyInjection` instead</RN>
<RN type="feature">Named/keyed document store support â€” `AddDocumentStore("name", opts => ...)` registers stores as .NET keyed singletons. Inject with `[FromKeyedServices("name")]` or resolve dynamically via `IDocumentStoreProvider.GetStore("name")`</RN>
<RN type="feature">Multi-tenancy support â€” two isolation strategies via `Shiny.DocumentDb.Extensions.DependencyInjection`: **shared-table** (single database with automatic `TenantId` column filtering) and **tenant-per-database** (separate database per tenant via lazy factory)</RN>
<RN type="feature">`ITenantResolver` interface â€” implement to provide the current tenant ID. Used by both multi-tenancy strategies to auto-resolve tenant context per request</RN>
<RN type="feature">`AddDocumentStore(configure, multiTenant: true)` â€” shared-table multi-tenancy registration. Adds a dedicated `TenantId` column and index to the schema; all queries are automatically filtered by the current tenant</RN>
<RN type="feature">`AddMultiTenantDocumentStore(Func<string, DocumentStoreOptions>)` â€” tenant-per-database registration. Each tenant gets a lazily-created separate database. `IDocumentStore` is registered as scoped and resolves to the correct tenant automatically</RN>
<RN type="feature">`TenantIdAccessor` on `DocumentStoreOptions` â€” core pipeline hook for shared-table multi-tenancy. When set, all queries include a `TenantId` filter and all inserts include the tenant value. A dedicated column and index are created automatically</RN>
<RN type="feature">New `Shiny.DocumentDb.IndexedDb` package â€” IndexedDB provider for Blazor WebAssembly with `IndexedDbDocumentStore`. Zero native dependencies, persists to the browser's IndexedDB via JS interop</RN>
<RN type="feature">SQLite WASM compatibility â€” `SqliteDatabaseProvider` now skips WAL pragma on `OperatingSystem.IsBrowser()`, spatial `R*Tree` is disabled in WASM, and `Backup()` is marked `[UnsupportedOSPlatform("browser")]`</RN>
<RN type="feature">New `Shiny.DocumentDb.LiteDb` package â€” LiteDB provider with `LiteDbDocumentStore`</RN>
<RN type="feature">New `Shiny.DocumentDb.CosmosDb` package â€” Azure Cosmos DB provider with `CosmosDbDocumentStore`</RN>
<RN type="feature">Spatial/geo query support â€” `WithinRadius`, `WithinBoundingBox`, and `NearestNeighbors` methods on `IDocumentStore` with default `NotSupportedException` for unsupported providers</RN>
<RN type="feature">`GeoPoint` readonly record struct â€” represents a WGS84 coordinate, serializes as GeoJSON `{"type":"Point","coordinates":[lng,lat]}`</RN>
<RN type="feature">`GeoBoundingBox` readonly record struct for area-based spatial queries</RN>
<RN type="feature">`SpatialResult<T>` wrapper â€” returns documents with computed `DistanceMeters` from the query center point</RN>
<RN type="feature">`MapSpatialProperty<T>` on `DocumentStoreOptions` â€” register which `GeoPoint` property to use for spatial indexing per document type</RN>
<RN type="feature">SQLite spatial support via `R*Tree` virtual tables â€” automatic sidecar table creation and CRUD sync for spatial-indexed documents</RN>
<RN type="feature">CosmosDB spatial support via native `ST_DISTANCE` and `ST_WITHIN` GeoJSON queries with automatic spatial index policy</RN>
<RN type="feature">`SupportsSpatial` property on `IDocumentStore` â€” check if the current provider supports spatial queries at runtime</RN>
<RN type="feature">`SqliteDocumentStore.ClearAllAsync()` â€” deletes all documents across all tables in the SQLite database, including spatial sidecar tables</RN>
<RN type="feature">Optimistic concurrency via document-level version properties â€” `MapVersionProperty<T>(x => x.RowVersion)` on all provider options classes. Version is set to 1 on insert, checked and incremented on update/upsert. Throws `ConcurrencyException` on mismatch. Stored inside the JSON blob â€” zero schema changes required</RN>
<RN type="feature">AOT-safe `MapVersionProperty<T>` overload â€” `MapVersionProperty<T>(string propertyName, Func<T, int> getter, Action<T, int> setter)` for trimming-safe deployments</RN>
<RN type="feature">`ConcurrencyException` â€” new exception type with `TypeName`, `DocumentId`, `ExpectedVersion`, and `ActualVersion` properties for diagnosing version conflicts</RN>

## 1.0 - March 6, 2026
<RN type="feature">Schema-free JSON document storage on top of SQLite</RN>
<RN type="feature">Mandatory typed `Id` property on document types (`Guid`, `int`, `long`, or `string`) â€” stored in both the SQLite column and the JSON blob so query results always include it</RN>
<RN type="feature">Auto-generation of Ids on Insert: `Guid` â†’ `Guid.NewGuid()`, `int`/`long` â†’ `MAX(CAST(Id AS INTEGER)) + 1` per TypeName. String Ids must be set explicitly â€” Insert throws for default string Ids</RN>
<RN type="feature">LINQ expression queries translated to json_extract SQL with support for equality, comparisons, logical operators, null checks, string methods, nested properties, and collection queries</RN>
<RN type="feature">Fluent query builder (IDocumentQuery) â€” chain .Where(), .OrderBy(), .OrderByDescending(), .GroupBy(), .Paginate(), .Select() and terminate with .ToList(), .ToAsyncEnumerable(), .Count(), .Any(), .ExecuteDelete(), .ExecuteUpdate(), .Max(), .Min(), .Sum(), .Average()</RN>
<RN type="feature">Pagination via .Paginate(offset, take) â€” translates to SQL LIMIT/OFFSET</RN>
<RN type="feature">Expression-based ordering â€” .OrderBy(u => u.Age) and .OrderByDescending(u => u.Name) on the fluent query builder</RN>
<RN type="feature">SQL-level projections via .Select() using json_object for extracting specific fields without full deserialization</RN>
<RN type="feature">IAsyncEnumerable streaming via .ToAsyncEnumerable() â€” yield results one-at-a-time without buffering</RN>
<RN type="feature">Expression-based JSON indexes for up to 30x faster queries on indexed properties</RN>
<RN type="feature">Full AOT and trimming support â€” all JsonTypeInfo parameters are optional and auto-resolve from configured JsonSerializerContext</RN>
<RN type="feature">Scalar aggregates: .Max(), .Min(), .Sum(), .Average() as terminal methods on the query builder</RN>
<RN type="feature">Aggregate projections with automatic GROUP BY via Sql.Count(), Sql.Max(), Sql.Min(), Sql.Sum(), Sql.Avg() marker methods</RN>
<RN type="feature">Collection-level aggregates in projections: Sum, Min, Max, Average on child collections (e.g. o.Lines.Sum(l => l.Quantity))</RN>
<RN type="feature">Explicit Insert / Update / Upsert API â€” Insert throws on duplicate Ids, Update throws if not found, Upsert deep-merges via json_patch</RN>
<RN type="feature">SetProperty â€” update a single scalar JSON field via json_set without deserializing the document. Supports nested paths</RN>
<RN type="feature">RemoveProperty â€” strip a field from the stored JSON via json_remove. Works on any property type</RN>
<RN type="feature">Typed Id lookups â€” Get, Remove, SetProperty, and RemoveProperty accept the Id as `object` (Guid, int, long, or string). Unsupported types throw ArgumentException</RN>
<RN type="feature">Bulk delete via query builder â€” .Where(predicate).ExecuteDelete() returns count of deleted documents</RN>
<RN type="feature">Bulk update via query builder â€” .Where(predicate).ExecuteUpdate(property, value) updates a property on matching documents via json_set() and returns count updated</RN>
<RN type="feature">Transactions with automatic commit/rollback via RunInTransaction</RN>
<RN type="feature">Hot backup via store.Backup(path) â€” copies the database to a file using the SQLite Online Backup API while the store remains usable</RN>
<RN type="feature">Dependency injection registration via AddSqliteDocumentStore</RN>
<RN type="feature">Configurable type name resolution (ShortName or FullName)</RN>
<RN type="feature">UseReflectionFallback option for strict AOT enforcement</RN>
<RN type="feature">SQL logging callback via DocumentStoreOptions.Logging</RN>
<RN type="feature">Raw SQL query and streaming support via store.Query(whereClause) and store.QueryStream(whereClause)</RN>
