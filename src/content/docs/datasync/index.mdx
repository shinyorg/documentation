`Shiny.Data.Sync` is a **reliable, background-capable, bidirectional JSON sync engine** between a mobile / desktop app and an HTTP backend. Where `Shiny.Net.Http` moves **files** in the background, `Shiny.Data.Sync` moves **records**: Create / Update / Delete operations queued in a local outbox, drained against your REST API, with a complementary inbox for delta pulls.

It mirrors the platform tier guarantees of `Shiny.Net.Http`:

| Platform | Outbox transport | Inbox transport | Survives app kill |
|---|---|---|---|
| iOS / Mac Catalyst | Background `NSURLSession` upload task | Background `NSURLSession` download task | **Yes** (both directions) |
| Android | Foreground Service + `HttpClient` | `HttpClient` (in-process) | Outbox **yes** (foreground notification visible); inbox no |
| Windows / Linux / macOS / base .NET | `HttpClient` + connectivity loop | `HttpClient` + connectivity loop | No — resumes on next launch |
| Blazor WebAssembly | `HttpClient` + `LocalStorage` | `HttpClient` + `LocalStorage` | No — sync runs while the tab is open |

## Why Shiny.Data.Sync?

**Your server, your rules.** No proprietary protocol, no special database tables, no middleware. If your API can return JSON and accept JSON, you're ready to go. Each entity has its own URL and can target a completely different backend.

**True background sync.** On iOS / Mac Catalyst the outbox **and** the inbox ride the same background `NSURLSession`, so queued operations and pending pulls survive the app being killed. On Android the outbox runs inside a foreground service; on every other target an in-process `HttpClient` loop reacts to connectivity events.

**AOT and trim safe.** All serialization runs through `Shiny.Json.Default` (the shared `ISerializer` from `Shiny.Extensions.Serialization`). Tag your entities with `[ShinyJsonContext]` and a source-generated module initializer wires them into the shared chain before any code runs — no per-endpoint `JsonTypeInfo` plumbing.

**One stream of everything.** Outbox queueing, retries, conflicts, inbox deltas, and tombstone deletes all surface through the unified `Activity` event. Subscribe once and you have your telemetry, your status bar, and your pull-to-refresh feedback.

## Core Concepts

- **Entities implement `ISyncEntity`** — one property: `Identifier`. No base classes, no attributes.
- **Endpoints are registered up-front** — one URL, one entity type, one stable key. Persisted with each queued operation, so renaming an endpoint key strands existing work.
- **Outbox** — queued `Create` / `Update` / `Delete` operations with serialized JSON payloads, retry counters, and exponential backoff.
- **Inbox** — server-driven delta pulls keyed by an opaque cursor. The default REST transport expects `{ cursor, hasMore, items: [{ id, verb, payload }] }`.
- **Direction** — per endpoint: `Both` (default), `PullOnly`, `PushOnly`.
- **Conflicts** — when the server returns 409 / 412, the delegate decides: accept remote, keep local, or merge.
- **Tombstones, soft-delete, expiry** — three layered strategies for removing entities the client should no longer have. See [Removal Strategies](/datasync/removal-strategies).

## Setup

<LibBuilder componentName="datasync" client:load />

### 1. Define an entity

Any record (class, struct, or record) that exposes a stable `Identifier`:

```csharp
using Shiny.Data.Sync;

public record TodoItem(string Identifier, string Title, bool Completed) : ISyncEntity;
```

### 2. Wire serialization (AOT-safe)

Add the entity to a `[ShinyJsonContext]`-attributed `JsonSerializerContext`. The source-generated module initializer wires it into `Shiny.Json.Default` before any sync code runs.

```csharp
[ShinyJsonContext]
[JsonSerializable(typeof(TodoItem))]
public partial class AppJsonContext : JsonSerializerContext;
```

The same context covers every endpoint that uses these types — no per-endpoint plumbing.

### 3. Implement the delegate

```csharp
using Shiny.Data.Sync;

public class MyDataSyncDelegate : IDataSyncDelegate
{
    public Task OnSent(SyncOperation op, string? responseBody)
    {
        // The server accepted the op. responseBody may carry a server-assigned id, ETag, etc.
        return Task.CompletedTask;
    }

    public Task OnError(SyncOperation op, int statusCode, Exception ex)
    {
        // The op exhausted its retry budget. Persist to a dead-letter store, surface to the user, or re-queue.
        return Task.CompletedTask;
    }

    public Task OnReceived(SyncReceivedItem item)
    {
        // The engine already deserialized item.Entity using Shiny.Json.Default.
        if (item.Entity is TodoItem todo)
            myLocalStore.Apply(todo, item.Verb);
        return Task.CompletedTask;
    }

    public Task<ConflictResolution> OnConflict(SyncOperation op, string remotePayload)
        => Task.FromResult(ConflictResolution.AcceptRemote);
}
```

### 4. Register services

```csharp
using Shiny;
using Shiny.Data.Sync;

builder.Services.AddDataSync<MyDataSyncDelegate>(opts =>
{
    opts.RegisterEndpoint<TodoItem>("https://api.example.com/todos");

    opts.RegisterEndpoint<Project>("https://api.example.com/projects", ep =>
    {
        ep.Direction = SyncDirection.Both;
        ep.Batch = true;                                  // coalesce queued ops per round-trip
        ep.UseMeteredConnection = false;                  // wait for WiFi
        ep.MinPullInterval = TimeSpan.FromMinutes(5);     // throttle scheduled pulls
        ep.DefaultConflictPolicy = ConflictPolicy.ServerWins;
    });
});

// Optional: a single cross-cutting auth header for every sync request
builder.Services.AddSyncInterceptor<MyAuthInterceptor>();

// Centralize base address / Polly handlers on the named client
builder.Services.AddHttpClient(RestSyncTransport.HttpClientName, c =>
    c.BaseAddress = new Uri("https://api.example.com")
);
```

On Blazor WebAssembly call `AddBlazorDataSync<TDelegate>` from `Shiny.Data.Sync.Blazor` instead — it persists the outbox and cursors to `LocalStorage` and drives sync from an in-process `HttpClient` loop on connectivity events.

```csharp
builder.Services.AddBlazorDataSync<MyDataSyncDelegate>(opts =>
{
    opts.RegisterEndpoint<TodoItem>("https://api.example.com/todos");
});
```

### 5. Queue and pull

```csharp
public class TodosService(IDataSyncManager sync)
{
    public Task Create(TodoItem item) => sync.Queue(SyncVerb.Create, item);
    public Task Update(TodoItem item) => sync.Queue(SyncVerb.Update, item);
    public Task Delete(TodoItem item) => sync.Queue(SyncVerb.Delete, item);

    public Task Refresh(CancellationToken ct = default) => sync.PullNow<TodoItem>(ct);
}
```

`PullNow<T>` bypasses the per-endpoint `MinPullInterval` — use it for pull-to-refresh. `PullAll` runs every endpoint that is `Both` or `PullOnly` and honours the throttle. A `SyncJob` is auto-registered, so periodic pulls run in the background without any additional plumbing.

## Observing sync

```csharp
sync.PendingCountChanged += (s, count) => StatusLabel.Text = $"{count} pending";

sync.UpdateReceived += (s, result) =>
{
    if (result.State == SyncOperationState.Error)
        ShowToast($"Sync failed: {result.Exception?.Message}");
};

sync.PullCompleted += (s, c) =>
{
    if (c.Error != null) ShowToast($"Pull failed for {c.EndpointKey}: {c.Error.Message}");
    else if (c.ItemsReceived > 0) ShowToast($"{c.ItemsReceived} new {c.EndpointKey} items");
};

// Unified stream — covers every outbox + inbox lifecycle moment
sync.Activity += (s, evt) =>
    Console.WriteLine($"{evt.Type} {evt.EndpointKey} items={evt.ItemCount} status={evt.StatusCode}");
```

The `Activity` event fires `SyncEvent` records for: `OutboxQueued`, `OutboxStarted`, `OutboxSent`, `OutboxFailed`, `OutboxConflict`, `OutboxRetryScheduled`, `OutboxCanceled`, `InboxPullStarted`, `InboxItemReceived`, `InboxPullCompleted`, `InboxPullFailed`, `TombstonesApplied`.

## Cancelling work

```csharp
await sync.Cancel(operationId);   // one operation
await sync.CancelAll<TodoItem>(); // everything for one endpoint
await sync.CancelAll();           // entire outbox (in-flight inbox pulls are left alone)
```
