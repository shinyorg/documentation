## v4.0 - April 11, 2026

<RN type="feature">**`INavigationBuilder`** ‚Äî fluent builder for multi-segment Shell navigation URIs. Chain `Add<TViewModel>()`, `Add(string)`, and `PopBack()` calls, then execute with a single `Navigate()`. Supports relative, root (`fromRoot: true`), and mixed pop/push patterns like `../../Page1/Page2`</RN>
<RN type="feature">**`INavigator.CreateBuilder(bool fromRoot)`** ‚Äî factory method to create an `INavigationBuilder` instance. Pass `fromRoot: true` to build absolute URIs prefixed with `//`</RN>
<RN type="feature">**Generated `INavigationBuilder` extensions** ‚Äî source generator now produces `NavigationBuilderNavExtensions.g.cs` with typed `Add{Name}()` methods for each `[ShellMap]` ViewModel, enabling fluent chains like `.AddDetail(id: 42).AddModal().Navigate()`</RN>
<RN type="enhancement" breaking={true}>**`relativeNavigation` parameter** ‚Äî `NavigateTo(string)` and `NavigateTo<TViewModel>()` now accept `bool relativeNavigation = true`. When `false`, the URI is prefixed with `//` for root navigation. Replaces the previous `SetRoot<TViewModel>()` method</RN>
<RN type="enhancement">**Generated `NavigationExtensions`** ‚Äî all generated `NavigateTo{Name}()` methods now include a `bool relativeNavigation = true` parameter</RN>

### Migration from v3.x

- `SetRoot<TViewModel>()` has been removed ‚Äî use `NavigateTo<TViewModel>(relativeNavigation: false)` instead
- Calls to `NavigateTo` that pass tuple args as positional parameters may need the `args:` named parameter to disambiguate from the new `bool relativeNavigation` parameter

```csharp
// Before (v3.x)
await navigator.SetRoot<HomeViewModel>(vm => vm.Message = "Hello");
await navigator.NavigateTo("details", ("Id", 42));

// After (v4.0)
await navigator.NavigateTo<HomeViewModel>(vm => vm.Message = "Hello", relativeNavigation: false);
await navigator.NavigateTo("details", args: [("Id", 42)]);

// New: Navigation Builder
await navigator
    .CreateBuilder()
    .AddDetail(id: 42)
    .AddModal()
    .Navigate();

// New: Pop back and push
await navigator
    .CreateBuilder()
    .PopBack(2)
    .AddHome()
    .Navigate();
```

## v3.2.1 - April 8, 2026

<RN type="fix">**MainThread dispatch on macOS & Linux** ‚Äî `MauiMainThread` now bypasses `MainThread.InvokeOnMainThreadAsync` on macOS and Linux, where MAUI's implementation fails or deadlocks. `INavigator` and `IDialogs` dispatch calls now work reliably on both platforms</RN>
<RN type="feature">**`IMainThread` interface** ‚Äî thread-marshalling abstraction used internally by `ShellNavigator` and `ShellDialogs`, now registered as a singleton so you can inject it directly instead of using `Microsoft.Maui.ApplicationModel.MainThread`</RN>
<RN type="feature">**`ShellServices` record** ‚Äî convenience aggregate of `INavigator`, `IDialogs`, and `IMainThread` ‚Äî inject a single parameter when a ViewModel needs most of them</RN>
<RN type="feature">**`UseDialogs<TDialog>()`** ‚Äî plug in a custom `IDialogs` implementation via `UseShinyShell(x => x.UseDialogs<MyDialogs>())`. The default `ShellDialogs` registration now uses `TryAddSingleton` so user overrides always win</RN>

## v3.2 - April 1, 2026

<RN type="feature">**ShinyShell base class** ‚Äî new `ShinyShell` class that overrides `OnNavigated` to deterministically set the initial page's BindingContext via Shell's own lifecycle, eliminating a race condition where the `Application.PageAppearing` event could fire before the handler was registered</RN>
<RN type="fix">**BindingContext inheritance fix** ‚Äî BindingContext checks now correctly detect inherited values (e.g., the Shell instance propagated down the visual tree) instead of only checking for null, ensuring ViewModels are always assigned to their mapped pages</RN>

### Migration from v3.1

Your `AppShell` (and any other Shell subclass) must now inherit from `ShinyShell` instead of `Shell`:

**AppShell.xaml** ‚Äî change root element:
```xml
<shiny:ShinyShell
    x:Class="MyApp.AppShell"
    xmlns="http://schemas.microsoft.com/dotnet/2021/maui"
    xmlns:x="http://schemas.microsoft.com/winfx/2009/xaml"
    xmlns:shiny="clr-namespace:Shiny;assembly=Shiny.Maui.Shell"
    xmlns:local="clr-namespace:MyApp"
    Title="MyApp">
    <!-- ... -->
</shiny:ShinyShell>
```

**AppShell.xaml.cs** ‚Äî change base class:
```csharp
using Shiny;

public partial class AppShell : ShinyShell
{
    public AppShell()
    {
        InitializeComponent();
    }
}
```

## v3.1.1 - March 17, 2026

<RN type="fix">`INavigationConfirmation` no longer triggers during programmatic navigation ‚Äî `CanNavigate()` is now only called for user-initiated navigation (back button, tab switches), not when navigating via `INavigator` methods</RN>

## v3.1 - March 11, 2026

<RN type="feature">`SwitchShell` ‚Äî swap the entire active Shell at runtime via `INavigator.SwitchShell(shell)` or `INavigator.SwitchShell<TShell>()` (DI-resolved). Fires `Navigating`/`Navigated` events with `NavigationType.SwitchShell` and respects `INavigationAware.OnNavigatingFrom`</RN>
<RN type="feature">`SwitchShell<TShell>()` ‚Äî generic overload resolves the Shell from the DI container, enabling constructor-injected Shell instances</RN>
<RN type="feature">`NavigationType.SwitchShell` ‚Äî new enum value for tracking shell switches in navigation events and analytics</RN>

## v3.0 - March 11, 2026

<RN type="feature">**IDialogs interface** ‚Äî new dedicated dialog service with `Alert`, `Confirm`, `Prompt`, and `ActionSheet` methods, injected separately from `INavigator` for clean separation of concerns</RN>
<RN type="feature">`Prompt` dialog ‚Äî display a text input dialog with customizable keyboard type, placeholder, initial value, and max length</RN>
<RN type="feature" githubNumber="1" pr={true} contributor="b4j4" repo="mauishell">`ActionSheet` dialog ‚Äî display a multi-option action sheet with cancel and destructive action support</RN>
<RN type="enhancement" breaking={true}>`Alert` and `Confirm` moved from `INavigator` to `IDialogs` ‚Äî improves testability and follows interface segregation</RN>

### Migration from v2.x

- `INavigator.Alert()` and `INavigator.Confirm()` have been removed ‚Äî inject `IDialogs` instead
- Replace `navigator.Alert(...)` with `dialogs.Alert(...)` and `navigator.Confirm(...)` with `dialogs.Confirm(...)`
- Add `IDialogs` to your ViewModel constructor parameters where dialogs are used
- `IDialogs` is automatically registered by `UseShinyShell()` ‚Äî no additional setup required

```csharp
// Before (v2.x)
public class MyViewModel(INavigator navigator)
{
    await navigator.Alert("Error", "Something went wrong");
    bool ok = await navigator.Confirm("Delete?", "Are you sure?");
}

// After (v3.0)
public class MyViewModel(INavigator navigator, IDialogs dialogs)
{
    await dialogs.Alert("Error", "Something went wrong");
    bool ok = await dialogs.Confirm("Delete?", "Are you sure?");

    // New capabilities
    var name = await dialogs.Prompt("Name", "Enter your name");
    var choice = await dialogs.ActionSheet("Options", "Cancel", null, "Edit", "Share");
}
```

## v2.2

<RN type="feature">Configurable source generation ‚Äî disable route constants via `ShinyMauiShell_GenerateRouteConstants` or navigation extensions via `ShinyMauiShell_GenerateNavExtensions` MSBuild properties (empty/missing = enabled, `false` = disabled)</RN>
<RN type="feature">Route-based naming ‚Äî the `route` parameter in `[ShellMap]` now drives the generated constant name and navigation method name (e.g., `[ShellMap<HomePage>("Dashboard")]` ‚Üí `Routes.Dashboard`, `NavigateToDashboard`)</RN>
<RN type="feature">Invalid route diagnostic ‚Äî **SHINY001** compiler error when the route value is not a valid C# identifier (hyphens, spaces, leading digits)</RN>
<RN type="enhancement">`AddGeneratedMaps()` is always generated ‚Äî even before any `[ShellMap]` attributes exist ‚Äî so you can wire up `MauiProgram.cs` immediately</RN>
<RN type="enhancement">`AddGeneratedMaps()` now uses inline string literals instead of `Routes.*` constants, so it works correctly even when route constant generation is disabled</RN>
<RN type="enhancement">When no route is specified, the generated name falls back to the page type name without the `Page` suffix (e.g., `[ShellMap<HomePage>]` ‚Üí `Routes.Home`)</RN>

### Migration from v2.1

- Route constant names may change if you specified explicit routes ‚Äî e.g., `Routes.Home` for `[ShellMap<HomePage>("Dashboard")]` is now `Routes.Dashboard`
- Navigation extension method names change similarly ‚Äî `NavigateToHome` becomes `NavigateToDashboard`
- Routes with invalid C# identifiers (hyphens, spaces, leading digits) now produce compile errors ‚Äî rename them to valid identifiers
